# OnMyAgent 应用创意指南

## 目标

把 OnMyAgent 官方能力、本地宿主实测行为、项目机制、产出物自动装载能力，以及不同目标用户和利益相关者的典型交流场景，沉淀成可检索、可复用、可审校的创意卡库。

这份指南服务两个目的：

- 作为行业场景研究员的创意库与能力来源。
- 作为后续“场景补位卡 -> 作战路径 -> 人机混编任务表 -> 3 天试点行动包 -> 唯一下一步”生成链路的上游灵感池。

## 证据优先级

创意库中的能力描述按下面顺序定性：

1. OnMyAgent 官方文档与更新日志。
2. 当前 OnMyAgent 宿主真实行为。
3. 当前 OnMyAgent 项目工作区、artifact-index、任务与预览落地证据。
4. 来自同一轮研究的归纳模式。

如果某个创意卡只来自归纳模式，没有官方文档或宿主证据支撑，`source_evidence.client_use_status` 只能是 `internal_reference_only` 或 `needs_human_verification`。

## 能力认识分层

创意库不要把所有能力混成一个平面。至少分成三层：

1. `overview_capabilities`
   指简介、快速开始、任务、对话、结果等页面明确承认的核心能力。
2. `changelog_confirmed_capabilities`
   指更新日志证实已经上线，但概览页未系统归纳的能力，例如专家中心演进、企业自建 Skill、自动化模板、Bot 推送、安全中心、Deeplink、分享能力。
3. `local_observed_combinations`
   指当前宿主真实跑通过的组合，例如 HTML/PDF 自动装载、ImageGen 配图、show_widget 摘要卡、宿主能力全景图、官方文档锚定的组合策略矩阵。

第二层和第三层可以进入创意库，但默认属于增强层或方法层，不得回写成“首值前提”。

## 能力分层

### 首值主线

这些能力可作为无连接器首值的一部分：

- 自然语言发起任务与持续追问。
- 本地文件读取、整理、转换、生成。
- 项目工作区沉淀任务、素材、模板、交付物。
- 结果区查看产物、全部文件、变更、预览。
- 文档、表格、报告、PPT、PDF、HTML 等产出物交付。
- 多任务并行与任务续作。

### 可选增强

这些能力可以提升附加值，但不是首值前提：

- 产出物自动装载交付环：Markdown、HTML、PDF、图片、内联摘要卡、loaded preview。
- 宿主能力全景分层与结果导向选路。
- 专家、技能、灵感库、资料库复用。
- 自动化模板、定时任务、Bot 推送、小程序推送。
- 助理、远程遥控、移动端和小程序协作。
- 云端部署、外链分享、深链和外部浏览器预览。

## 高绩效使用策略

### 线下交流

- 会前：把客户画像、历史材料、议程目标整理成会前简报和三问清单。
- 会中：用 OnMyAgent 当前会话补位材料，必要时通过助理/移动端形成远程补位。
- 会后：把纪要、录音摘要、照片、任务清单转成 3 天试点包或责任矩阵。

### 线上交流

- 会前：把网页快照、活动页、报表、FAQ、回放文本转成可视化诊断材料。
- 会中：用 loaded preview、PDF、HTML 和摘要卡支撑共享审阅。
- 会后：将结论沉淀为项目模板、自动化草案、FAQ、AB 参数包或内容补更日历。

### 利益相关者交流

- 管理层：强调“一个工作流缺口、一个唯一下一步、一个试点周期”。
- 跨部门：强调 RACI、依赖图、风险边界和 owner。
- 客户/合作方：强调可验收交付物、证据账本和承诺边界。
- 法务/合规：强调敏感句预警、证据缺口与人工审批点。

## 时机三段式

创意卡优先按 `会前 -> 会中 -> 会后` 设计：

- 会前：降本。用一页纸、摘要卡、问题清单、预读材料压缩进入成本。
- 会中：聚焦。用 HTML、看板、SVG、变更视图、预览页支撑决策或讨论。
- 会后：闭环。用 PDF、动作包、责任矩阵、自动化草案、项目沉淀推进下一步。

如果某张卡无法说明三段中的主阶段，就还不够成熟。

## 格式与认知负荷

不要默认“文档越长越专业”。格式要匹配读者认知负荷：

- 管理层和投资人优先一页纸、摘要卡、看板。
- 客户决策者优先 HTML 方案页或短版 PDF。
- 执行层优先 HTML/PDF 动作包与清单。
- 内部协作优先 Markdown、项目工作区、可追溯变更。

创意卡里的 `format_strategy` 应明确说明这一点。

## 设计规则

- 每张创意卡必须只有一个 `unique_next_step`。
- 每张创意卡必须声明 `persona_anchor`，让卡片绑定一个具体目标用户和主要利益相关者。
- 每张创意卡必须声明 `timing_pattern`，取值为 `pre_meeting`、`in_meeting`、`post_meeting` 或 `full_cycle`。
- 每张创意卡必须声明 `format_strategy`，说明为什么选择一页纸、HTML、PDF、Markdown、摘要卡或 loaded preview。
- 每张创意卡必须声明 `evolution_evidence_note`，把能力来源分到 `overview_capabilities`、`changelog_confirmed_capabilities` 和 `local_observed_combinations`，并重申 host-local 与 service-consume 边界。
- 任何探索性增强路线都不能替代最终唯一 CTA。
- `assistant_remote_optional`、`mini_program_push_optional`、`connector_optional` 只能作为可选增强族。
- 自动化默认只允许 `PAUSED` 候选，不允许默认为 `ACTIVE`。
- 任何 loaded preview、预览打开、present_files、文件写入都只属于 `host_local`。

## 政策方向映射

当输入材料是最新部委政策、联合行动计划、官方征集通知或权威转载时，不要把创意卡写成政策摘要器。标准做法是先做一层映射，再生成场景卡：

1. 政策方向：
   提炼文件里被反复强调的场景方向、适用行业、关键角色、基础设施要求、数据要求和安全边界。
2. 场景缺口：
   把政策方向翻译成一个可验证的工作流缺口，例如“门店智能化运营提效”“中小企业经营管理自动化”“能源高价值场景开放与验证”。
3. 试点动作：
   默认收敛到 3 天试点包、1 个唯一下一步和 1 份证据账本，不把“政策支持”直接写成“业务已经成立”。

政策方向类卡片至少补充这些内容：
- `trigger_conditions`：
  必须说明是“新政策发布”“政策窗口期”“申报/对接/立项前”还是“企业内部战略研讨”触发。
- `inputs.required`：
  必须同时包含“政策原文或权威转载”和“企业当前流程/材料/数据”。
- `source_evidence`：
  至少 1 条 `official_doc`，并标明 `internal_reference_only` 或 `needs_human_verification`，除非用户已明确要求对客引用。
- `reuse_notes`：
  说明这张卡是“政策方向 -> 场景映射”还是“政策方向 -> 申报/试点执行”。

推荐输出类型：
- `policy_fit_matrix`
- `scene_mapping_card`
- `regulatory_readiness_note`
- `three_day_pilot_packet`
- `policyDirectionLedger`

边界：
- 政策方向只是一种上游证据，不等于产品承诺、成交闭环或官方背书。
- 未结合企业真实流程、数据和角色前，不要把政策里的大方向直接写成用户当前最优缺口。

## 推荐使用顺序

1. 先用创意卡锁定一个最适合当前用户的场景。
2. 再把卡片映射到五件套首值输出。
3. 如果用户要求更强分发、更强协作或更强复用，再激活可选增强。
4. 如果进入项目沉淀阶段，再把高频创意卡升级成模板或专家封装。

## 明确不做的事

- 不把连接器、服务侧工具、Bot、助理、小程序、移动端写成首值前提。
- 不把预览、分享、loaded preview 当成客户已阅读或 continued use。
- 不把本地等效封装写成官方上架证明。
- 不把创意库变成“功能目录”或“工具百科”。
