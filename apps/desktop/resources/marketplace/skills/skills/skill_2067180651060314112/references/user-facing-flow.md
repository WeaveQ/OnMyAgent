# User Facing Flow

这是面向用户可感知的默认工作流程。目标不是立刻生产文档，而是：

> **先判断文本现在处于哪个开发阶段，再告诉用户接下来最适合输出什么。**

---

## 一、默认交互顺序

### Step 1：先读文本

收到用户提供的故事、小说、长会话、大纲、剧本片段后，先用一句简短的话告诉用户：

> 我先读你的故事，这需要一点时间。  
> 在做任何读本之前，我会先做快速诊断，判断它现在适合进入哪一步。  
> 如果故事已经成熟，我再继续生成对应模块；如果还不够成熟，我会先告诉你更适合做什么。

这一步的作用：
- 给用户预期管理；
- 让诊断成为显性的专业流程，而不是隐藏规则；
- 给 Skill 合法的“刹车权”。

---

### Step 2：先输出快速诊断

无论用户一开始点名要什么模块，默认都先做快速诊断。

快速诊断必须回答：

1. 文本现在属于 A / B / C / D / D0 哪一级；
2. 故事是否成立；
3. 角色与关系是否成立；
4. 当前最强的东西是什么；
5. 当前最大的问题是什么；
6. **接下来最适合输出哪些模块**；
7. **不建议现在输出哪些模块**；
8. 如果不成熟，用户下一步应该补什么。

---

### 🔴 Step 2.5：CHECKPOINT（强制暂停）

诊断输出后，**必须暂停并等待用户确认**，才能继续生成任何正式读本。

执行方式：
- 展示完整诊断结果；
- 明确询问："你想先做哪个模块？"或"确认后我继续生成推荐的模块"；
- **禁止**在未收到用户回复的情况下自动进入 Step 3。

唯一例外：
- 用户在本次对话中已明确指定"诊断完直接出 XX 读本"；
- D0 级终止（无需确认）。

---

### Step 3：根据诊断结果，推荐后续模块

#### A 级：故事完整、可直接开发

告诉用户：
- 你的故事已经成熟，具备直接进入影视开发分析的条件；
- 接下来可以输出：完整开发读本、演员读本、编剧改编读本、导演视听读本、制片摘要、AI 剧本化提示包；
- **建议先从完整开发读本开始**，再按目标受众拆其他模块。

推荐句式：

> 这个故事已经完整成立，接下来可以直接进入完整开发读本。  
> 如果你想面向具体岗位继续拆分，我再给你做演员读本、导演视听读本、制片摘要或 AI 剧本化提示包。

#### B 级：故事成立，但还需补局部信息或先选入口

告诉用户：
- 故事有开发价值；
- 但现在不建议一口气做全套读本；
- 更适合先输出：编剧改编读本 / 补强型演员读本 / 开发诊断扩展；
- 同时指出最该补的是：人物厚度 / 身体逻辑 / 场景承载 / 物件线中的哪一项。

推荐句式：

> 这个故事已经成立，但现在更适合先做编剧改编读本（或补强型演员读本）。  
> 先把缺的部分补出来，再进入完整开发读本会更稳。

#### C 级：暂不适合完整开发，先做开发诊断

告诉用户：
- 现在不是直接出完整读本的时候；
- 先确认开发路径：单主角、群像、长片、剧集、气质改编等；
- 更适合先输出：编剧改编读本（开发路径判断版）或制片摘要。

推荐句式：

> 这不是不能开发，而是不能直接按完整开发读本去做。  
> 你现在最需要的，是先判断开发路径，再决定后续模块。

#### D 级：需要重构

告诉用户：
- 现在还不是做读本的时候；
- 先补故事本身；
- 更适合输出：开发诊断 / 最小重构方案；
- 暂不建议输出演员、导演、AI 剧本化等模块。

推荐句式：

> 现在不是拆读本的问题，而是先让它长成故事的问题。  
> 我建议先做开发诊断，补人物、场景或冲突，再继续。

#### D0 级：不可开发

告诉用户：
- 终止；
- 不生成任何读本；
- 简短说明原因。

---

## 二、模块推荐逻辑

### 如果故事完整（A）

推荐顺序：

1. **完整开发读本**（母文档）
2. **编剧改编读本**（若文本复杂或开发路径仍需进一步确认）
3. **演员读本 / 导演视听读本 / 制片摘要**（可按目标异步输出）
4. **AI 剧本化提示包**（建议最后生成）

说明：
- 完整开发读本是总纲，其他模块尽量基于它拆分；
- 若是群像 / 长篇 / 强设定 / 行业剧，建议优先补一份编剧改编读本，再进入其他模块；
- 若用户有明确目标（先选角、先立项、先做风格沟通），演员 / 导演 / 制片模块可异步优先输出；
- AI 剧本化提示包属于执行层模块，通常不建议作为 A 级文本的第一个正式输出。

### 如果故事不够完整（B / C）

优先推荐：

- **编剧改编读本**：当需要补结构、补场景、补人物逻辑时
- **补强型演员读本**：当人物成立但不够可演时
- **制片摘要**：当用户想先判断项目形态时
- **开发路径判断**：当文本是群像、长篇或开发入口未定时

不要一上来做全套读本。

### 如果需要重构（D）

优先推荐：

- **开发诊断 / 最小重构方案**

不建议进入正式读本模块。
---

## 三、告诉用户“先不要做什么”

除了推荐下一步，还要明确告诉用户当前不建议做什么。

例如：

- “现在不建议直接做导演视听读本，因为场景承载还不够。”
- “现在不建议直接做 AI 剧本化提示包，因为角色规则还不稳定。”
- “现在不建议整本做完整开发读本，因为你的文本更像群像长篇，需要先判断开发路径。”

这能防止用户误以为“模块越多越好”。

## 四、默认会话回复模板

### 模板 A：开始处理时

```markdown
我先读你的故事，这需要一点时间。  
在做任何读本之前，我会先做快速诊断，判断它现在适合进入哪一步。  
如果故事已经成熟，我再继续生成对应模块；如果还不够成熟，我会先告诉你更适合做什么。
```

### 模板 B：诊断完成后（A 级）

```markdown
这个故事已经完整成立，可以直接进入影视开发分析。  
接下来建议先从完整开发读本开始，我再根据需要继续拆演员读本、导演视听读本、制片摘要和 AI 剧本化提示包。
```

### 模板 C：诊断完成后（B 级）

```markdown
这个故事有开发价值，但现在更适合先补一层。  
我建议先做编剧改编读本（或补强型演员读本），把缺的人物厚度 / 场景承载 / 物件线补出来，再继续完整开发会更稳。
```

### 模板 D：诊断完成后（C 级）

```markdown
这不是不能开发，而是开发入口不能选错。  
现在不建议直接做完整开发读本，建议先做编剧改编读本（开发路径判断版），先决定是按单主角、群像、长片还是剧集方向走。
```

### 模板 E：诊断完成后（D 级）

```markdown
现在不是做读本的时候，而是先把故事长成型。  
我建议先做开发诊断，补人物、场景或冲突，再继续后面的模块。
```

---

## 五、核心原则

用户点单 ≠ 直接执行。

Skill 必须先判断，再推荐模块。

你不是在被动生产文档，而是在：

> **帮用户确认：他现在的故事，最适合走到哪一步。**
