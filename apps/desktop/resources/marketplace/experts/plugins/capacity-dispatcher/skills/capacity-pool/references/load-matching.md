# 多因素配载匹配

把「待调度订单」与「当前可用运力」做成 1–3 套可执行方案，并写清推荐理由与风险。有经验的调度会综合 **顺路、载重、司机意愿、时效、空驶**，在脑子里快速比较几套方案后再决定；本方法把比较过程显式化。

## 输入要求

### 待调度订单（每票至少）

| 字段 | 说明 | 缺失时 |
|------|------|--------|
| 装货地 / 卸货地 | 至少到区/镇 | 追问，否则无法算顺路与空驶 |
| 货物吨/方 | 至少一项 | 无法校核载重 |
| 时效窗 | 最晚装/到时间或档位 | 按普通时效并标注假设 |
| 车型/特殊要求 | 尾板、冷藏、危化等 | 无则按普通车 |

选填：件数、可否拼车、指定司机、装卸要求、货值。

### 可用运力

优先使用当前运力池（fresh / aging）；用户粘贴的清单同样可解析。**stale 默认不入选**，除非调度明确「仍有效」。

## 匹配因子（默认权重，可按场景调整）

| 因子 | 权重参考 | 好 | 差 / 否决 |
|------|----------|----|-----------|
| 顺路 / 方向一致性 | 高 | 装货在司机前进方向或当前附近 | 明显逆向、大绕行 |
| 载重 / 方数适配 | 高（硬约束） | 剩余 ≥ 需求，利用率适中 | 超载/超方 → **否决** |
| 时效可达 | 高（硬约束） | freeAt + 路程能进窗 | 明显赶不上 → **否决或标高风险** |
| 空驶距离 | 中高 | 空驶短、可顺路带货 | 空驶过长抬成本 |
| 司机意愿 / 方向偏好 | 中 | 方向在 preferredLanes 或 any | 明确不接该方向 → 降权或剔除 |
| 车型与能力 | 中高（部分硬） | 尾板/冷链等命中 | 硬性缺失 → **否决** |
| 信息新鲜度 | 中 | fresh | aging 降权并注明；stale 默认否 |
| 习惯与表现 | 低（仅有依据） | 常跑该线、响应快 | 无数据不编造 |

> 权重不是打分游戏：输出用自然语言写清「为什么选 A 不选 B」，避免假精确总分。

## 方案生成规则

1. **先硬过滤**：超载超方、硬性车型/资质不符、明确不接方向、stale 未确认、明显赶不上时效 → 剔除或仅列入「不可用说明」。
2. **再软排序**：顺路 + 空驶 + 意愿 + 新鲜度 +（有依据的）习惯。
3. **输出 1–3 套方案**（不是一长串候选人列表）：
   - **方案 A（首选）**：综合最优，写清核心理由。
   - **方案 B（备选）**：在空驶/时效/司机意愿上取不同权衡。
   - **方案 C（可选）**：仅在有实质差异时给出（如拼载、分拆、次日发）。
4. **一车多票 / 多车一票**：
   - 可拼则给拼载顺序与剩余吨方；
   - 必须拆则说明拆票逻辑与风险；
   - 禁止为「凑方案」建议超载。
5. **高风险必须单列**，并写「建议你最终确认」：
   - aging 信息未复核、时效极紧、空驶偏长、司机意愿不明、危化/冷链边界、可能压车/二次过磅争议等。

## 方案输出模板

```markdown
## 配载推荐（基于当前运力，{时间}）

### 订单摘要
| 票 | 线路 | 吨/方 | 时效 | 特殊要求 |
|----|------|-------|------|----------|

### 方案 A（推荐）
- **运力**：{车牌/司机} | {位置} | {车型} | 余{吨/方} | 新鲜度
- **怎么配**：{单票直送 / 与某票拼载 / 卸完接…}
- **推荐理由**：
  1. 顺路/空驶：…
  2. 载重：…
  3. 时效：…
  4. 意愿/习惯：…（无则写「未提供，未加权」）
- **风险/待确认**：… 或「无明显高风险」

### 方案 B（备选）
…

### 未入选运力（简述）
| 运力 | 未选原因 |
|------|----------|

### 需要你拍板
- [ ] 锁定方案 A / B / C
- [ ] 是否接受 aging 运力 / 较长空驶 / 紧时效
```

结构化 JSON 建议字段：`schemes[]`（id, rank, assignments[], rationale[], risks[]）、`rejected[]`、`needsConfirmation[]`。

## 与运力池的关系

| 场景 | 动作 |
|------|------|
| 仅有运力汇报 | 只维护/查询池，不硬做配载 |
| 仅有订单、无运力 | 列出缺运力字段，给群点名话术，不编造车辆 |
| 订单 + 运力 | 走完整多因素匹配，出 1–3 方案 |
| 用户已有池，追加订单 | 以池为底做匹配；匹配后可提示「若锁定请更新该车状态」 |

## 铁律

- 不编造运力位置、剩余载重、空闲时间来「凑」漂亮方案。
- 不输出鼓励超载、无资质危化、明显超时疲劳的方案。
- 方案是建议，**派单承诺与对外通知由调度最终确认**。
