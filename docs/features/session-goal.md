# 会话目标生命周期设计

## Status

**ACCEPTED design (2026-07-10).** Implementation largely landed; this doc remains the product/behavior contract for goal lifecycle.

## 目标

让“追求目标”、规划和访问权限成为按会话隔离的运行能力；目标模式在发送前可见，发送后进入可暂停、继续和恢复的目标运行视图。

## 根因

当前实现同时使用 `kind` 与 `pursueGoal` 表示协作模式，并且目标运行面板仅在代码类别、已生成 `goal_intent` 运行时的条件下展示。模式选择、运行时创建和展示条件没有统一的规范化入口，导致用户可看到“追求目标”按钮但没有对应的目标视图。

## 设计

- 保持已有 `ComposerCollaborationMode` 数据形状，不新增服务端 schema：`{ planning: false, pursueGoal: true }` 是唯一的目标模式表示；`craft` 不再隐式表示追求目标。
- 统一模式解析：目标不依赖助理类别；代码、办公和专家会话都可进入目标流程。
- 会话内目标预览：选择追求目标但尚未发送时，在输入框上方显示“等待任务目标”；首次发送后用该输入创建目标运行时。
- 运行时面板：目标、当前检查点、待办、耗时、状态、暂停/继续/结束操作都来自该 `sessionId` 的运行时记录。
- 规划模式：继续保持“先生成计划、再确认执行”，且与目标状态互斥。
- 访问权限：默认访问逐项请求；完全访问只自动批准当前运行时会话，不写入其他会话。
- 隔离约束：所有状态读写以真实 `sessionId`（新会话发送前使用 `draft:<workspaceId>`）为键；创建后仅把该草稿键迁移到新 session，不能覆盖或清空同工作区的其他会话。

## 验收标准

1. 选择追求目标后，发送前能看到目标预览。
2. 首次发送后，代码、办公和专家会话都显示目标运行面板。
3. 目标运行时的暂停/继续/清除只影响当前 session。
4. 刷新后同一 session 的目标、协作模式和访问策略恢复；其他 session 保持各自状态。
5. 规划运行时和目标运行时不同时显示。
